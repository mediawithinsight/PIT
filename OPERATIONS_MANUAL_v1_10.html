0. The one rule everything else follows

The FILE is the source of truth. Storage is a working surface.

  • SEED_POSITIONS, PRICES, ATR, HIGH_WATER, INTEL_DATA — all live in the artifact file. Durable, versioned, identical on every device.
  • IndexedDB holds only what you type — entries, targets, staged sells, core locks, cash.
  • Storage may be evicted, may be wiped, may be device-local. It is never allowed to be the only copy of anything that matters.

Consequence: book facts (shares, avg cost, price, stop) are stripped from storage on every load. The file always wins on those. That is deliberate.

1. Data workflows

1.1 Monday–Thursday — daily prices

You send: closing prices as a table. Claude updates: PRICES block in RBS + INTEL_DATA.positions.prices in PIT. Recomputed automatically: market values, allocation %, R:R, Chandelier stops, range position. Not recomputed: ATR, volume states, relative strength, 20-day range bounds. Those need the workbook.

1.2 Friday — the full refresh

You send: Historic Volume and Price ... .xlsx (the extended workbook). Claude updates: | Block | Source | |---|---| | PRICES | Close column, latest row | | ATR | Average True Range column | | RANGE20 | 20-session high/low | | VOL (5-state) | Volume level + trend | | RS | vs GDXJ benchmark | | HIGH_WATER | New highs, if any ticker printed one |

This is the only day the derived layers move. Mid-week, they are stale by design — consistently stale, which is safe. Mixed-date data is not.

1.3 When you make a trade

You send: ticker, account, buy/sell, quantity, price. Claude updates BOTH tools: - RBS: SEED_POSITIONS shares/avgCost · SEED_CASH · clears executed staging - PIT: INTEL_DATA.positions.lots · asOf date

Expect an amber stale strip on that card afterward. That is correct — see §2.2.

Claude does NOT update PRICES to your fill price. One ticker marked to today while nine are marked to Friday gives a portfolio with mixed as-of dates. It corrects itself on the Friday refresh.

1.4 Before you send anything — export first

  1. Open RBS → scroll to bottom → GENERATE
  2. COPY → paste into chat alongside your prices/trades
  3. Claude bakes your typed state into the next build

Why: the values you typed live in IndexedDB, which Claude cannot see. Without the export, a rebuild would ship you a file that has forgotten your entries, targets, and staged sells.

1.5 After every rebuild

Upload the new file to project knowledge. Otherwise the project copy goes stale and future chats work from an old version — the exact failure this project exists to prevent.

1.6 Account roles — the structural decision (14 Jul 2026)

The two accounts are not interchangeable. They have different jobs.

Account Role Behaviour
RRSP Core account High-conviction positions held through the thesis. Add on thesis-aligned dips. Sell only on thesis-change or genuine overweight — never out of fear of a further drop. Some swing trading allowed, but the core positions stay.
TFSA Swing account Where catalyst-timed trades actually happen. Rotation, trims, re-buys, catalyst plays.

Why this matters for the tools: a "trim" recommendation on an RRSP core position and a "trim" on a TFSA swing mean two completely different things. The merged tool should eventually read account role, because position-blind advice can tell you to swing-trade a position that is deliberately core. Until then, you apply the filter: is this RRSP-core? then the bar to sell is thesis-change, not price.

This is a decision that was circled for weeks and landed on the 14 Jul drive. It belongs here because it changes how every recommendation should be read.

2. What every flag means

2.1 Storage badge (header, always visible)

Badge Meaning Action
🟢 STORAGE ON — no changes to save yet IndexedDB alive, nothing typed yet None
🟢 ALL CHANGES SAVED · 14:32 Last successful write, with time None. The timestamp is the proof.
🔴 NOT SAVING — [reason] Writes are failing Do not trust the session. Export what you can and reload.

Known: window.storage is dead on iOS. IndexedDB works on both. If the badge is red on desktop, something else is wrong.

2.2 Amber strip — "Book changed" (per card)

Means: shares or avg cost for this ticker changed since you typed the inputs on this card. Your numbers were reasoned against a position that no longer exists.

Fires when: a trade is applied, or new book data is baked in.

RE-ANCHOR resets: amount to add · sell qty · sell price · confirm boxes RE-ANCHOR keeps: entry · target · stop (price levels — thesis, not quantity) · core shares (your decision, only clamped if the new share count makes it impossible)

Decision logged 14 Jul 2026 — core = one field, clamped to shares held. The idea of letting core exceed shares held (as an accumulation target / floor to rebuild toward — e.g. "I want 100k RDU RRSP") was considered and rejected for now. Core stays a single field meaning "of what I hold, this much is locked." A target is a different concept (drives buys; core constrains sells) and would need its own field — not being built. The RRSP↔TFSA relocation that prompted it stays a decision made outside the tool. Do not re-open without a new reason. RE-ANCHOR never touches: shares · avg cost · price. Those come from the file.

No undo. It asks first and lists exactly what it will change.

2.3 Red strip — "Levels don't add up" (per card)

Means: arithmetic, not judgement. One of: - Target ≤ entry → reward is negative, R:R cannot compute - Stop ≥ entry → risk is negative, R:R cannot compute

The deeper reading (§2.3 of Trading Log): if the trailing stop sits above your intended add level, your stop fires before your entry is ever reached. The plan contradicts itself. This is not a display bug — it is the tool telling you the plan is incoherent.

It never flags a target for being ambitious. CANX at 71x triggers nothing. That is a judgement call and it is yours.

2.4 ATR REVIEW badge

Means: price has fallen below the Chandelier trailing stop (high-water − 3 × ATR).

This is a REVIEW TRIGGER, NOT A SELL SIGNAL. Go and look at the catalysts, filings, and macro. The stop does not make the decision.

2.5 Volume states (5)

State Meaning
SURGE Level high, trend rising
BUILDING Level rising off a base
FADING Level falling — liquidity leaving. You may not be able to exit later.
DORMANT Almost no volume
NORMAL Nothing to see

FADING is the one that costs money. It is not about price — it is about whether the exit door is still open.

2.6 20-day range bar (under Entry)

Shows where price sits between its 20-session low and high. - Amber ≥80% — upper end of range - Blue ≤20% — lower end - Grey — middle

This is a question generator, not a signal. "Low = good, high = bad" is the mean-reversion read and it is wrong for a catalyst-driven trader. The question is: does the range position agree or disagree with what the catalyst timeline says should be happening right now?

2.7 Derived fields (greyed, dashed border)

Stop is computed live from ATR + high-water on every load. It cannot be typed into. A manual stop would be silently overwritten on the next load — worse than not being able to type one at all.

If you disagree with a stop, the fix is the high-water anchor or the ATR, not the field.

3. Who owns which number

Number Owner Changes when
Shares, avg cost The file A trade is applied

ATR, RANGE20, volume, RS The file Friday workbook refresh

If a number is in the wrong column, that is a bug. The stop field pretending to be editable was exactly this, and it was fixed.

4. Known gaps and cautions

  • RDU + RAK pre-2026 transaction history is missing. Share reconciliation goes negative. Last-buy dates are used as an approximation. RAK's high-water anchor ($0.190, 12 Mar) rests on this incomplete data — the stop is arithmetically right but the anchor may not be.
  • CRCL high-water is $0.140 but the stock traded at $0.150 on 14 Jul. Anchor needs updating. (open)
  • Full 2025 transaction history would unlock realized P&L and "since last flat" high-water anchoring.
  • Storage is device-local. Phone and desktop each hold their own IndexedDB. They do not sync. The export→bake cycle is what keeps them aligned. MCP sync was considered and deliberately rejected — it solves a problem that does not exist at current usage.
  • Throwaway/test builds must use a different DB name. A test build once wrote a fake book stamp into the production database. Any experimental build should be treated as radioactive until proven otherwise — check the version badge before tapping anything.
  • RBS New Book scenario buys at ENTRY when entry is set (v2.13.4), falling back to market price when entry is 0. So the "shares this buys" and blended avg cost model a fill at your planned entry, not at the current quote. The row is tagged (entry) or (mkt) so it's explicit. R:R in the scenario is computed from the same buy price.

5. The environment (hard-won, do not re-derive)

Four browser APIs fail silently inside the Claude artifact sandbox. All four were assumed to work and none did:

API Status
localStorage / sessionStorage Banned everywhere
window.storage Works on desktop. Dead on iOS — every verb returns "Invalid response format"
navigator.clipboard Blocked — throws where nothing is listening
confirm() / prompt() Blocked — returns nothing at all, so a button just appears dead

IndexedDB works. Tested on iOS: survives session close, artifact reopen, and new artifact versions. It is the transport.

The rule that came out of this: test the platform, do not infer it. Every one of the four above failed quietly, and quiet failure is indistinguishable from success right up until it costs you.

6. Update requests — the four standing workflows

6z. Summary Intelligence refresh — refresh-by-conversation (there is NO button)

The Summary Intelligence card (Macro Context, Insider Moves & Alpha, Ranked Trade Opportunities, Scenario Matrix) is a baked research snapshot in INTEL_DATA. It does not auto-update when a ticker is added, and it cannot — insider transactions, macro copper/gold context, and alpha theses are not in the filings, and PIT's sandbox has no network. There is no import box and no export button in PIT (both were removed; export/import lives in RBS for user state only).

The only way to refresh it is to ask Claude in chat. Workflow: you say "refresh the intel" → Claude runs the research pass (SEDI/insider filings, macro, per-name alpha) → compiles fresh INTEL_DATA for review → bakes it in directly (same two-message pattern as a ticker add). Trigger: the red coverage banner on the Summary card ("covers N of M tickers · missing: …") and the per-section dates. When the banner is red or the dates are aging, ask for a refresh. Adding a ticker and refreshing intel are decoupled — add fast (mechanical), refresh intel later (research). The banner keeps them honestly linked so neither drifts silently.

On Request 3 — why it cannot be automatic

Claude has no SEDAR access. Every filing in project knowledge is there because you uploaded it. There is no path by which a new MD&A reaches the tools on its own.

The consequence, stated plainly: if a filing drops and you do not upload it, PIT's Filings Intelligence keeps showing the old picture — and it will look completely current. Nothing will indicate otherwise. This is the same failure mode as every other bug in this project: wrong data that does not announce itself.

What the diff should report (the delta, not a re-read): - New warrants issued · warrant overhang change - Runway shift (quarters of cash remaining) - Going-concern language appearing or disappearing - New debt - Dilution since last filing - Any change to project ownership / lease vs. title - Corporate obligations lane — re-parse the commitments note (see §11). This is the one that silently drifts, so it is a required step of every filing diff, not an optional extra.

Trigger: CALENDAR — every 90 days. Decided 14 Jul 2026.

Chosen over event-based deliberately: event-based relies on you noticing a filing hit, and "you have to remember" is exactly the failure mode this project keeps engineering out. A 90-day clock fires whether or not you were watching.

First clock starts from the filing dates already in project knowledge (interim/annual filings dated Mar–Jun 2026). The Guide tab computes days-since per ticker and raises a red dot past ~100 days — a 10-day grace so the flag means "go look", not "you're late".

This does not stop you filing early. If you see one hit, upload it — the clock is a floor, not a ceiling.

On feature 3 — the classification layer (✅ BUILT v2.15.0)

The 11-stage pipeline (§8.3) is per project. A company holds several projects at different stages. The Explorer/Developer/Producer band is a company-level roll-up — derived from the furthest-advanced project, never hand-set (or it goes stale the moment a project advances).

Mapping as built: | Band | Rolls up from | |---|---| | Explorer | TARGET · DRILLING · ASSAYS · MRE — and standalone PERMITTING/CONSTRUCTION with no economic study (= drill permits) | | Developer | An economic study exists: PEA · PFS · FS (permitting/construction only elevate with a study) | | Producer | PRODUCTION | | (DORMANT ignored for banding — orthogonal) |

The decided seam: the Explorer→Developer line is the first economic study (PEA). The critical clause — permitting/construction without a study stays Explorer — is what keeps CANX (Arizona drill permits, no NI 43-101 resource) an Explorer instead of a mis-banded Developer. Confirmed against data 14 Jul 2026.

In-progress studies — the orange WIP dot: a stage underway but not yet published (e.g. CGNT's PEA, targeted Q4 2026) carries an orange dot on the top-right of its stage pill, and the same dot rides on the DEVELOPER header pill when the band qualifies only through an in-progress study. The band flips to DEVELOPER while the pill still tells the truth — the study isn't published. A published study at the same level clears the dot. Schema: profile.stagesWip: [] — a subset of stages, orthogonal (isMilestone pattern), additive; absent means nothing in progress.

Prospect generator = a flag, not a band. Business model, orthogonal to lifecycle. Stored as "isProspectGenerator": true on the ticker object; renders as a PROSPECT GEN pill. Set on RDU (Radius Gold — the archetype). Royalty/Streamer: none in the basket, not built.

Drill vs mine permitting — the flag (v2.15.1). The permitting stage is ambiguous: a drill permit is an explorer signal, a mine permit is a developer signal. Schema: profile.permittingType: "drill" | "mine". When an active permitting pill has a type set, it renders the label suffix (DRILL) or (MINE) plus an orange flag on the top-right. This makes the pill self-explaining and reinforces the banding rule — a drill permit with no economic study stays Explorer. Set on CANX (drill — Arizona BLM drill permits). Absent = plain PERMITTING pill, no flag.

Current basket: 9 Explorers · 1 Developer (CGNT, WIP) · 0 Producers · RDU flagged PROSPECT GEN · CANX permitting typed DRILL.

8. PIT reference — schema, stages, design rules

8.1 Record schema

Authored fields: n · symbol · project · title · category · confidence · date · estDate · sourceType · link · notes

Boolean weights — orthogonal; one record can carry several:

Flag Icon Meaning
isMilestone ★ milestone event
obligation $ cash the company must pay out
risk ! adverse development
warrant W warrant expiry tranche {shares, strike}
isNew newly added record
loggedAt date Claude added it (separate from date/estDate)

5 categories only: News · Forecasted Catalyst · Employee · Event · Research

CORE PRINCIPLE: Milestone-ness is a WEIGHT, not a category. Same for obligation / risk / warrant. This is why they survive lifecycle changes — a forecast obligation stays an obligation after it is paid; a milestone stays a milestone after it happens.

8.2 Computed at render — never stored

The single most important rule in the build. Every item below was once a stored flag that went stale and produced a bug.

Computed Rule
visualState past / future / milestone / media / result ← category + dates
recOverdue estDate < today && !date
isFreshNew isNew && loggedAt within 14 days (auto-expires, no button)
moneyness live quote vs warrant strike → ITM / OTM
freshness badge days since last CONFIRMED date; stale >14d; future dates ignored
triage buckets ≤45d Active / 46–90d Watching / else Hold
Gantt horizon futureMonths is a FLOOR, not a ceiling — auto-extends

NEW_WINDOW_DAYS = 14 · STALE_DAYS = 14

This is the same principle that drives the User Guide tab (§7). A flag computed at render cannot go stale. A stored one always can.

8.3 Stage pipeline — 11 stages, MULTI-FLAG

A project carries every stage currently true (e.g. Mocoa = MRE + DRILLING + PEA):

TARGET → DRILLING → ASSAYS → MRE → PEA → PFS → FS
       → PERMITTING → CONSTRUCTION → PRODUCTION → DORMANT

NI 43-101 is deliberately NOT a stage. It is the reporting standard governing MRE/PEA/PFS/FS disclosure, not a step in the pipeline.

8.4 Going-concern levels

Level Badge Meaning
funded ✓ green filing states program is fully funded
none ✓ green no going-concern qualification
note ⚠ orange note-level disclosure
auditor ⚠⚠ red flagged in the audit opinion itself

8.5 Design rules

  • Colour means STATUS (teal = confirmed, rose = risk/overdue, amber = new, navy = warrant). Never spend colour on chrome like open/closed state.
  • Icons for FACTS; interpretation in the TEXT. $ and ! work because nobody has to agree with a judgement call about whether an obligation exists. A "good news / bad news" icon would launder opinion as data.
  • Confidence badges are non-negotiable: SOURCED (found via live search) · SYNTHESIZED (Claude's interpretation of sourced data) · SPECULATIVE (forecast with material uncertainty)
  • Section IDs on every card (SUM:macro-context, ANDC:filings-intelligence) so issues can be referenced precisely.
  • Recommendations are labelled FRAMEWORK, NOT A DIRECTIVE — something to argue with, not follow.

8.6 Structural rules

  • Two durable data blocks: BEGIN_TICKER_DATA/END_TICKER_DATA and BEGIN_INTEL_DATA/END_INTEL_DATA
  • The version const must live OUTSIDE the markers — a migration script ate it once.
  • Artifact-only environment: in-artifact API calls do not work (proxy rejects web_search responses; CORS blocks external hosting). All research runs through chat, then gets baked into the file.

9. Forecast catalyst timing — the duration model

Origin: the original vision (build-chat, project inception) was that forward catalysts get an estimated landing date so they position on the Gantt and feed the triage-bucket math, even when the company gives no date — "it's all speculation, but the view gives a real-time picture." This is that estimating made explicit instead of ad-hoc.

Where it stands (audited 15 Jul 2026): 81 of 98 forward catalysts carry an estDate and sit on the Gantt. 17 are pure TBD (no estDate) — 12 of them RDU (Jonco permit chain, Amalia, Holly, Tierra Roja option payments). So the approach is followed ~83%, but was never systematic: each estimate was a one-off judgement, and messy projects (Jonco) defaulted to TBD instead of an estimate.

9.1 The rule

A forward catalyst's estDate = anchor event date + typical duration for that event type in that jurisdiction. The estimate is always confidence: Speculative (renders amber with a * on the Gantt, and as Est. <date>* in the list) so it never masquerades as a company-confirmed date.

9.2 Typical durations — planning estimates, NOT authoritative

Event Canada (BC/Yukon) Peru / LatAm Other
Assay turnaround (core → results) ~4–8 wks ~8–12 wks Namibia ~8–12 wks
Exploration/drill permit NoW ~4–12 wks FTA→CIRA→GREM chain ~6–18 mo AZ BLM Notice ~2–8 wks
Drill start after permit granted ~2–6 wks (mobilization) ~3–8 wks —
MRE after campaign completes ~3–6 mo ~3–6 mo —
PEA after MRE ~6–12 mo ~6–12 mo —
PFS after PEA ~12–18 mo ~12–18 mo —

9.3 The anchoring problem (why 12 RDU catalysts are still TBD)

An estimate can only be pinned if its anchor has a date. Jonco's chain — community access → environmental → MEM auth → permits granted → earn-in clock → drill start → assays — is anchored to the permit approvals, which have no filed date. You cannot pin a chain to nothing.

So there is a genuine choice for un-anchored chains, and it is the user's call: - (a) Floating estimate — anchor to today + typical duration, re-based each refresh. ✅ CHOSEN & BUILT v2.17.0. - (b) Leave TBD — kept only for genuinely indefinite events (a stalled JV with no trigger and no typical duration).

9.4 Floating estimates — as built (v2.17.0)

Schema: floating: true + floatDays: N on the catalyst (orthogonal, isMilestone pattern). estDate is computed at load = today + floatDays, so it re-bases every time the file opens and never goes stale. A floating estimate is never "overdue" (its date is always in the near future).

Visual — calm and distinct, deliberately NOT the red overdue treatment: - Gantt marker: slate, dotted border (vs amber-dashed for anchored estimates, vs rose-dashed for overdue). - Gantt date label: slate italic, prefixed ≈ (not the amber *). - List: ≈ Est. <date> in slate + a small dotted FLOATING tag. - Tooltip everywhere: "no confirmed anchor — slides each refresh."

Applied to: the 6-step Jonco permit chain (community access → environmental → MEM auth → permits granted → clock → drill start), floatDays 90–300.

Still pure TBD (deliberately): Amalia JV reactivation / resource / Palmillas payment, Holly JV review — indefinitely deferred, no typical duration to float against. Tierra Roja GREM auth + option-payment start — pending your call on floatDays.

The re-basing caveat (know this): a floating date that never gets closer is the signal that real news hasn't landed. If a floating catalyst has been sliding for months, that itself is information — it belongs on the Guide-tab watch list, not as a date you trust.

10. RBS share structure, normalized comparison & EV (v2.16.x)

What it's for: junior miners have wildly different share counts, so raw share price tells you nothing about size. A $0.10 stock with 500M shares is bigger than a $0.50 stock with 50M. This feature re-prices every ticker to one common share count so price becomes a clean size comparison, then strips out cash to compare on enterprise value.

10.1 The one input — header

"Normalized Share Count" field in the header (default 271,547,000 — your baseline; edit it and every card recomputes). Call it N. It's just a common yardstick; the exact number doesn't matter, only that it's the same for all tickers. Saved in IndexedDB.

10.2 The per-card strip — how to read each number

Each ticker card shows a purple strip:

Field Formula How to read it
Norm Price (price x issued shares) / N "What this stock would trade at if it had N shares." Because N is constant, this is a pure market-cap ranking — higher Norm Price = bigger company. Compare it across cards.
EV-adj (under Norm Price) (market cap - cash) / N Same idea, cash removed. The cash-adjusted size — the sharper number for developers sitting on a treasury.
Ratio vs Norm issued shares / N How lean or bloated the cap table is vs your baseline. < 1 = fewer shares than N (leaner); > 1 = more. A low ratio means any re-rate moves the price harder (fewer shares to spread it over); a high ratio dilutes the move.
Issued Shares from filings Basic issued & outstanding (not fully-diluted — see 10.4). The % cash sub-line is cash / market cap.
Footer — Mkt cap . Cash . EV, in dollars. EV = market cap - cash.

10.3 Worked example — RDU

Price $0.16 . 115.7M shares . N = 271.547M . cash $1.4M - Norm Price = 0.16 x 115.7M / 271.547M = $0.068 - Ratio = 115.7M / 271.547M = 0.43x (leaner than baseline) - Mkt cap = $18.5M, EV = 18.5 - 1.4 = $17.1M

10.4 What to remember (the caveats)

  • Basic shares, not fully-diluted. The normalization uses issued & outstanding (matches your Excel). Warrant/option overhang is not in this number — read it alongside PIT's warrant-overhang flags, which is where dilution risk lives.
  • Cash is as-of each filing's date (e.g. RDU Dec 31, FEO Mar 31), so a name that has since burned or raised will have drifted. CASH refreshes on the quarterly filings diff (S6, Request 3 / #3), not the Friday price refresh.
  • EV is the developer's number. For explorers with tiny treasuries, EV is about market cap and the distinction barely matters. For a funded developer (FEO 31% cash, KFR 21%, RAK 45%), EV can reorder the basket vs raw market cap — that's the whole point of showing it.
  • Norm Price is a comparison tool, not a valuation. It says nothing about whether a company is cheap — only how it ranks by size against the others. Cheap/expensive is the thesis + catalyst + balance-sheet work, not this strip.

10.5 Updating the data

Share counts and cash are baked from filings, one value per ticker (SHARES and CASH blocks in RBS). On a financing or a new quarterly filing, they change — hand them over in the filings diff and they get re-baked, same as any other filing-sourced number.

10.6 Burn rate & runway — the dilution-timing forecast (v2.17.0)

What it's for: runway = how many months until a company runs out of cash and must raise. A financing you see coming is a buying discipline (wait for it); a surprise raise tanks your entry. This turns the balance sheet into a raise-timing map.

The line on each card: Burn ~$X/mo · Runway ~Y · cash-out ~MMM 'YY, colour-coded — red < 6 mo, amber 6–12, green > 12.

How burn is defined: monthly cash consumed before financing = operating outflow + exploration/development outflow, excluding one-time investing inflows (option payments received, disposals). Taken from the most recent filing's cash-flow statement, annualized to monthly. This is why RDU and OGW aren't shown as near-zero burn — both had one-time investing inflows this period that would otherwise mask their real spend.

Runway = cash ÷ monthly burn. Cash-out date = filing date + runway.

What to remember: - Exploration is discretionary. A company can slow drilling to stretch runway, so a short runway is a prompt to check, not a certainty. Heavy drillers (ANDC, CGNT) show high burn but can throttle. - "Cash-out elapsed" means the filing-date runway has already passed — a raise has likely happened (and the baked cash is stale) or the company is overdue. Refresh the filing. - EST tag = the filing's cash-flow statement didn't parse cleanly and the burn is an estimate. Currently on RAK (annual filing) and CGNT (messy OCR) — verify and replace these two. - Burn is trailing, not forward. A company ramping drilling will burn faster than its last filing shows.

Current read: CRCL (elapsed) and OGW (~6 mo) are the near-term dilution risks; FEO and KFR just raised and are years out. The BURN block refreshes on the quarterly filings diff, same as SHARES/CASH.

11. Timeline lanes — catalyst vs. risk (v2.20.0)

The organizing rule: the two timelines answer two different investor questions. - Project timelines = catalyst-facing. "What could make this worth more?" Drilling, results, resource estimates, studies, permits-that-unlock. - Corporate timeline = risk-facing. "What could dilute or drain me?" Warrant expiries, financings, and every property/option payment obligation.

So obligations MOVE, they do not copy. A payment obligation comes off its project timeline and onto Corporate, keeping the project name in the label (e.g. "Horetzky option: 2M shares + $200K due"). Project lanes stay pure catalyst; the risk lane holds every forced cash-out and dilution event in one place. The Summary tab's Cash Obligations Schedule (§ built v2.19.0) is the cross-ticker roll-up of the same records.

The one judgment call: an event that is both a catalyst and an obligation — e.g. NLR's first NI 43-101 resource milestone, which is a value inflection that also triggers a share payment — stays on the project timeline as a catalyst. The dominant meaning wins. Its dilution shows up in the balance-sheet/warrant work, not as a risk-lane marker.

11.1 Filing-update review rules — REQUIRED every diff

Obligation schedules are the area that silently drifts (dates slip, elections resolve, tranches get paid). On every filing diff (§6, Request 3), re-parse the commitments note and update the Corporate lane:

Item type Rule
Dated tranche ("on or before Oct 15 2027") Plot at that date. If paid since last filing → mark done / remove.
Conditional ("earlier of 5,000m drilled or Oct 15 2028") Plot at the backstop date; note the trigger. If the trigger fired early, re-date to actual.
"OR" amount ("$300K or 4M shares") Show both — the election (cash-drain vs. dilution) is itself the risk. Once elected in a filing, collapse to what was actually paid.
Recurring (e.g. FEO 2% NSR advance royalty, $100K/yr until production) Plot the next 1–2 only, labelled "annual until production" — never infinite markers that bury the lane. Advance the window each filing.
Sentiment (Tier 2) Re-set each obligation’s bull/bear/neutral read against current terms + price + treasury. A tag from a prior filing may have flipped.

Why this can't be automatic: the same reason as everything else — Claude has no SEDAR access, and a stale obligation lane looks completely current. The commitments note must be re-read from each uploaded filing and the lane hand-updated. This is now a checklist line in the Request-3 diff so it isn't forgotten.

Update this document whenever a workflow changes. A manual that is out of date is worse than none — it makes wrong behaviour look sanctioned.
